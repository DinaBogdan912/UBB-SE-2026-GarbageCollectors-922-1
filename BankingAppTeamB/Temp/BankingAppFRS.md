# [cite_start]1. Introduction [cite: 79]

## [cite_start]1.1 High-Level User Flow Overview [cite: 80]
[cite_start]The following sequence describes the primary path a user takes through the application, from authentication to any functional module: [cite: 81]

[cite_start]Authentication Gate → Main Dashboard → Functional Modules → Action Execution → Confirmation/Feedback [cite: 82]

* **Step 1 – Authentication:** User arrives at the Login Page. [cite_start]Authenticates via credentials, OAuth, or 2FA. [cite: 83] [cite_start]On success, a session token is issued and the user is redirected to the Main Dashboard. [cite: 84]
* [cite_start]**Step 2 – Main Dashboard:** Displays a summary of the user’s cards, account balances, recent transactions, and quick-action shortcuts. [cite: 85] [cite_start]Serves as the navigation hub to all modules. [cite: 86]
* [cite_start]**Step 3 – Module Navigation:** User selects a module from the primary navigation (Transfers, Bills, Cards, History, Exchange, Savings & Loans, Investments, Statistics, Support Chat). [cite: 87]
* [cite_start]**Step 4 – Action Execution:** Within each module, the user performs specific operations (e.g., initiating a transfer, paying a bill). [cite: 88] [cite_start]Sensitive operations require re-authentication or 2FA confirmation. [cite: 89]
* [cite_start]**Step 5 – Confirmation:** The system provides immediate feedback (success/failure notification), updates the dashboard state, and logs the action in the transaction history. [cite: 90]

# [cite_start]2. Authentication and Access Control Module [cite: 91]
[cite_start]This module governs all entry points into the application and ensures that only verified users gain access to banking features. [cite: 92]

## [cite_start]2.1 Login Page (FR-AUTH-001) [cite: 93]
### [cite_start]2.1.1 Description [cite: 94]
The Login Page is the application’s entry point. [cite_start]It must present a clean, professional interface with the bank’s branding. [cite: 95] [cite_start]The page shall support three distinct authentication pathways. [cite: 96]

### [cite_start]2.1.2 Functional Requirements [cite: 97]

[cite_start]**FR-AUTH-001.1 – Credential-Based Login** [cite: 98]
* [cite_start]The system shall display a login form containing two input fields: Email Address (used as the unique username) and Password. [cite: 99]
* [cite_start]The Email field shall validate format compliance on the client side before submission. [cite: 100]
* [cite_start]The Password field shall mask input by default, with a toggle icon to reveal/hide the password. [cite: 101]
* [cite_start]Upon submission, the system shall authenticate the credentials against the user store. [cite: 102]
* [cite_start]On success, issue a session token (JWT or equivalent) and redirect to the Main Dashboard. [cite: 103]
* [cite_start]On failure, display a generic error message ("Invalid email or password") without revealing which field is incorrect. [cite: 104]
* [cite_start]After five consecutive failed login attempts from the same IP or for the same account, the system shall temporarily lock the account for 15 minutes and notify the user via their registered email. [cite: 105]

[cite_start]**FR-AUTH-001.2 – Third-Party OAuth Login** [cite: 106]
* [cite_start]The Login Page shall display clearly labelled buttons for "Sign in with Google" and "Sign in with GitHub". [cite: 107]
* [cite_start]Clicking a third-party button shall initiate an OAuth 2.0 authorization code flow, redirecting the user to the respective provider’s consent screen. [cite: 108]
* [cite_start]Upon successful OAuth callback, the system shall match the returned email to an existing account or create a new user profile . [cite: 109]
* [cite_start]The user is then redirected to the Main Dashboard. [cite: 110]
* [cite_start]If the OAuth provider returns an error or the user denies consent, the system shall display an informative error and return the user to the Login Page. [cite: 111]

[cite_start]**FR-AUTH-001.3 – Two-Factor Authentication (2FA)** [cite: 112]
* [cite_start]If the user has enabled 2FA in their profile settings, the system shall, after successful credential verification, redirect to a 2FA Challenge Screen instead of the Dashboard. [cite: 113]
* [cite_start]The 2FA Challenge Screen shall prompt the user to enter a 6-digit one-time password (OTP). [cite: 114]
* [cite_start]Supported 2FA methods shall include: time-based OTP via an authenticator app (e.g., Google Authenticator, Authy), and OTP delivered via SMS or email to the registered contact. [cite: 115]
* [cite_start]The user shall have the option to select their preferred 2FA method if multiple are configured. [cite: 116]
* [cite_start]OTP codes shall expire after 60 seconds for TOTP and 5 minutes for SMS/email OTP. [cite: 117]
* [cite_start]After three consecutive incorrect OTP entries, the session shall be invalidated and the user must restart the login process from the Login Page. [cite: 118]

## [cite_start]2.2 Forgot Password Flow (FR-AUTH-002) [cite: 119]
### [cite_start]2.2.1 Description [cite: 120]
[cite_start]This flow allows users who have forgotten their password to securely reset it via a verified email address. [cite: 121]

### [cite_start]2.2.2 User Flow [cite: 122]
* [cite_start]**Step 1:** User clicks "Forgot Password?" link on the Login Page. [cite: 123]
* [cite_start]**Step 2:** System displays a form requesting the registered email address. [cite: 124]
* **Step 3:** User submits their email. [cite_start]The system always responds with "If an account exists with this email, a reset link has been sent" (to prevent email enumeration). [cite: 125]
* [cite_start]**Step 4:** If the email exists, the system generates a one-time, time-limited (30 minutes) password reset token and sends a reset link to the email. [cite: 126]
* [cite_start]**Step 5:** User clicks the link, which opens a Set New Password page. [cite: 127] [cite_start]The form requires the new password (minimum 8 characters, at least one uppercase letter, one lowercase letter, one digit, and one special character) and a confirmation field. [cite: 128]
* [cite_start]**Step 6:** On successful password reset, the system invalidates all existing sessions for the account, displays a confirmation message, and redirects to the Login Page. [cite: 129]

# [cite_start]3. Main Dashboard Module [cite: 130]
[cite_start]The Main Dashboard is the central hub of the application. [cite: 131] [cite_start]It aggregates key account information and provides navigation to all functional modules. [cite: 132]

## [cite_start]3.1 Dashboard Layout (FR-DASH-001) [cite: 133]
### [cite_start]3.1.1 Description [cite: 134]
Upon successful authentication, the user lands on the Main Dashboard. [cite_start]The dashboard shall be organized into the following zones: [cite: 135]

[cite_start]**FR-DASH-001.1 – Navigation Bar (persistent across all pages)** [cite: 136]
* [cite_start]A top or side navigation bar shall be visible on every page of the application. [cite: 137]
* [cite_start]Navigation items shall include: Dashboard (Home), Transfers, Bill Payments, Cards, Transaction History, Currency Exchange, Savings & Loans, Investments & Trading, Statistics, Customer Support Chat, and Profile. [cite: 138]
* [cite_start]The navigation bar shall highlight the currently active module. [cite: 139]
* [cite_start]The bar shall display the user’s name or avatar, a notification bell icon with an unread count badge, and a Log Out button. [cite: 140]

[cite_start]**FR-DASH-001.2 – Cards Overview Panel** [cite: 141]
* [cite_start]The dashboard shall display all payment cards associated with the user’s accounts as interactive card components. [cite: 142]
* [cite_start]Each card component shall display: the card brand logo (Visa, Mastercard, etc.), the last four digits of the card number (remaining digits masked as ••••), the cardholder name, the expiry date (MM/YY format), and the associated account name and current balance. [cite: 143]
* [cite_start]Card data (full card number, CVV) shall be hidden by default. [cite: 144]
* [cite_start]A "Show Details" action on each card shall require re-authentication (password or biometric) before revealing full card data for a limited time (60 seconds), after which it auto-hides. [cite: 145]
* [cite_start]Cards shall be displayed in the user’s preferred sort order (see FR-CARD-003 for sorting preferences). [cite: 146]

[cite_start]**FR-DASH-001.3 – Quick Actions Panel** [cite: 147]
[cite_start]A quick actions area shall provide one-click shortcuts to: New Transfer, Pay a Bill, Exchange Currency, and View Full Transaction History. [cite: 148]

[cite_start]**FR-DASH-001.4 – Recent Transactions Summary** [cite: 149]
* [cite_start]The dashboard shall display the five most recent transactions across all accounts. [cite: 150]
* [cite_start]Each transaction entry shall show: date and time, counterparty name or merchant, amount and currency, transaction type (debit/credit), and a visual indicator for the status (completed, pending, failed). [cite: 151]
* [cite_start]A "View All" link shall navigate to the full Transaction History module. [cite: 152]

* [cite_start]The user shall be able to edit phone number and residential address directly. [cite: 157]
* [cite_start]Changes to the legal name or date of birth shall require a support ticket or KYC re-verification. [cite: 158]

## [cite_start]4.3 Notification Preferences (FR-PROF-003) [cite: 165]

# [cite_start]5. Card Management Module [cite: 168]
[cite_start]This module provides full lifecycle management of payment cards associated with the user’s accounts. [cite: 169]

## [cite_start]5.1 View Cards (FR-CARD-001) [cite: 170]
* [cite_start]The system shall list all active, frozen, and expired cards belonging to the user. [cite: 171]
* [cite_start]Each card entry shall display: card type (debit/credit/prepaid), card brand and last four digits, associated bank account name and IBAN, cardholder name, expiry date, and status (Active, Frozen, Expired, Cancelled). [cite: 172]
* [cite_start]The user shall be able to expand a card to view additional details: daily and monthly spending limits, contactless payment status (enabled/disabled), online transaction status (enabled/disabled), and ATM withdrawal limits. [cite: 173]

## [cite_start]5.2 Show / Hide Sensitive Card Data (FR-CARD-002) [cite: 174]
* [cite_start]By default, the full card number and CVV shall be masked. [cite: 175]
* [cite_start]The user may trigger "Show Full Card Details" which shall require re-authentication (password entry, biometric, or 2FA code). [cite: 176]
* [cite_start]Upon successful re-authentication, the system shall reveal the full 16-digit card number and CVV/CVC for a maximum of 60 seconds, then automatically re-mask the data. [cite: 177]
* [cite_start]Each reveal event shall be logged in the user’s security audit trail. [cite: 178]

## [cite_start]5.3 Sort Cards by Preference (FR-CARD-003) [cite: 179]
* [cite_start]The user shall be able to reorder cards via drag-and-drop or an explicit "Set as Primary" action. [cite: 180]
* [cite_start]The sort preference shall persist across sessions and be reflected on the Main Dashboard. [cite: 181]
* [cite_start]The user may sort cards by: custom order (default), balance (ascending/descending), or card type. [cite: 182]

## [cite_start]5.4 Card Operations (FR-CARD-004) [cite: 183]
* [cite_start]**Freeze/Unfreeze Card:** The user shall be able to temporarily freeze a card, instantly blocking all transactions. [cite: 184] Unfreezing shall restore normal functionality. [cite_start]Both actions require confirmation. [cite: 185]
* [cite_start]**Set Spending Limits:** The user shall be able to modify daily transaction limits, monthly spending caps, ATM withdrawal limits, and contactless payment limits. [cite: 186]
* [cite_start]**Enable/Disable Online Transactions:** The user shall be able to toggle online and contactless transaction capabilities per card. [cite: 187]
* [cite_start]**Delete (Cancel) Card:** The user shall be able to permanently cancel a card. [cite: 188] [cite_start]This action requires 2FA confirmation and a mandatory confirmation dialog explaining the irreversibility. [cite: 189] [cite_start]Cancelled cards remain visible in a “Cancelled” state for 90 days for reference, then are archived. [cite: 190]
* [cite_start]**Request New Card:** The user shall be able to request a new physical or virtual card, selecting the account to link, card type, and preferred currency. [cite: 191]

# [cite_start]6. Money Transfer Module [cite: 192]
[cite_start]This module enables the user to send money to other accounts, both within the bank and externally. [cite: 193]

## [cite_start]6.1 New Transfer (FR-TRANS-001) [cite: 194]
### [cite_start]6.1.1 User Flow [cite: 195]
* [cite_start]**Step 1 – Select Source:** User selects the originating account/card from a dropdown of their active accounts, showing the current available balance. [cite: 196]
* [cite_start]**Step 2 – Enter Recipient:** User enters the recipient details: recipient name, IBAN or account number, bank name (auto-populated from IBAN where possible), and an optional reference/description. [cite: 197]
* [cite_start]**Step 3 – Enter Amount:** User enters the transfer amount and selects the currency. [cite: 198] [cite_start]If the currency differs from the source account, the system displays the applicable exchange rate and converted amount in real time. [cite: 199]
* [cite_start]**Step 4 – Review:** The system displays a comprehensive summary: source account, recipient details, amount, applicable fees, estimated arrival date, and total debit amount. [cite: 200]
* **Step 5 – Confirm:** User confirms the transfer. [cite_start]For amounts exceeding a configurable threshold (e.g., €1,000), the system shall require 2FA confirmation. [cite: 201]
* [cite_start]**Step 6 – Result:** System displays a success screen with a transaction reference number, or an error screen with a specific error code and retry option. [cite: 202]

## [cite_start]6.2 Saved Beneficiaries (FR-TRANS-002) [cite: 203]
* [cite_start]The system shall allow the user to save recipient details as a "Beneficiary" for future transfers. [cite: 204]
* [cite_start]The user shall be able to view, edit, and delete saved beneficiaries. [cite: 205]
* [cite_start]When initiating a new transfer, the user shall be able to select from saved beneficiaries to auto-populate recipient fields. [cite: 206]

# [cite_start]7. Bill Payment Module [cite: 207]
[cite_start]This module provides the user with the ability to pay recurring and one-time bills directly from the application. [cite: 208]

## [cite_start]7.1 Pay a Bill (FR-BILL-001) [cite: 209]
### [cite_start]7.1.1 User Flow [cite: 210]
* [cite_start]**Step 1 – Select Biller:** The user searches for or selects a biller from a categorized directory (Utilities, Telecom, Insurance, Rent, Government, Subscriptions, Other). [cite: 211] [cite_start]The directory shall be populated via integration with a bill payment API provider. [cite: 212]
* [cite_start]**Step 2 – Enter Details:** User enters the biller-specific reference (e.g., customer/account number), the amount to pay (or selects “Pay in full” if the outstanding balance is retrievable via API), and selects the source account. [cite: 213]
* [cite_start]**Step 3 – Review and Confirm:** System displays a summary including biller name, reference, amount, source account, and applicable fees. [cite: 214] [cite_start]User confirms the payment. [cite: 215]
* [cite_start]**Step 4 – Result:** System processes the payment and displays confirmation with a receipt number. [cite: 216] [cite_start]The transaction is recorded in the Transaction History. [cite: 217]

## [cite_start]7.2 Saved Billers and Recurring Payments (FR-BILL-002) [cite: 218]
* [cite_start]The user shall be able to save frequently used billers for quick access. [cite: 219]
* [cite_start]The user shall be able to set up recurring automatic payments: selecting a biller, amount (fixed or “full balance”), source account, frequency (weekly, monthly, quarterly), and start/end dates. [cite: 220]
* [cite_start]The system shall send a notification 24 hours before each scheduled payment, and a confirmation notification upon execution. [cite: 221]
* [cite_start]The user shall be able to view, pause, modify, or cancel any recurring payment. [cite: 222]

# [cite_start]8. Transaction History Module [cite: 223]
[cite_start]This module provides a comprehensive, searchable record of all financial activity across the user’s accounts. [cite: 224]

## [cite_start]8.1 View Transaction History (FR-HIST-001) [cite: 225]
* [cite_start]The system shall display a chronological list (newest first by default) of all completed, pending, and failed transactions. [cite: 226]
* [cite_start]Each transaction entry shall display: date and timestamp, transaction type (transfer, bill payment, card payment, FX exchange, loan disbursement, investment trade, ATM withdrawal, etc.), counterparty or merchant name, amount and currency, debit or credit indicator, running balance after the transaction, and status (Completed, Pending, Failed, Reversed). [cite: 227]

## [cite_start]8.2 Search, Filter, and Sort (FR-HIST-002) [cite: 228]
* [cite_start]**Search:** Free-text search across merchant name, reference number, and description. [cite: 229]
* [cite_start]**Filters:** By date range, transaction type, amount range (min/max), specific account or card, status, and debit/credit direction. [cite: 230]
* [cite_start]**Sort:** By date (ascending/descending), amount (ascending/descending). [cite: 231]
* [cite_start]Filters and sort settings shall be combinable and persist during the session. [cite: 232]

## [cite_start]8.3 Transaction Details (FR-HIST-003) [cite: 233]
[cite_start]Clicking on any transaction shall open a detail view showing: all fields from the list view, the full originating and destination account/IBAN, any associated fees, the exchange rate applied (for FX transactions), and a downloadable PDF receipt. [cite: 234]

## [cite_start]8.4 Export (FR-HIST-004) [cite: 235]
[cite_start]The user shall be able to export transaction history for a selected date range in the following formats: PDF statement, CSV file, and Excel (.xlsx) file. [cite: 236]

# [cite_start]9. Currency Exchange Module [cite: 237]
[cite_start]This module enables users to convert funds between different currencies held in their accounts. [cite: 238]

## [cite_start]9.1 Exchange Currency (FR-FX-001) [cite: 239]
### [cite_start]9.1.1 User Flow [cite: 240]
* [cite_start]**Step 1:** User selects the source account and source currency. [cite: 241]
* [cite_start]**Step 2:** User selects the target currency from a list of supported currencies. [cite: 242]
* [cite_start]**Step 3:** User enters the amount to exchange (in source or target currency). [cite: 243] [cite_start]The system displays the real-time exchange rate (fetched from a rate provider API), the converted amount, any applicable commission or spread, and the final amount the user will receive. [cite: 244]
* **Step 4:** User reviews and confirms. [cite_start]The rate shall be locked for 30 seconds from the time of display. [cite: 245] [cite_start]If the user does not confirm within 30 seconds, the rate is refreshed. [cite: 246]
* [cite_start]**Step 5:** On confirmation, the system debits the source account in the source currency and credits the same or a designated target account in the target currency. [cite: 247] [cite_start]A confirmation screen displays the final details and transaction reference. [cite: 248]

## [cite_start]9.2 Exchange Rate Information (FR-FX-002) [cite: 249]
* [cite_start]The module shall display a rate ticker or table showing live exchange rates for popular currency pairs (EUR/USD, EUR/GBP, EUR/RON, etc.). [cite: 250]
* [cite_start]The user shall be able to set rate alerts: selecting a currency pair and a target rate, and receiving a push/email notification when the rate is reached. [cite: 251]

# [cite_start]10. Statistics and Analytics Dashboard [cite: 252]

* [cite_start]The user shall be able to select the time period for analysis: current month, last 3 months, last 6 months, last 12 months, custom date range. [cite: 256]
* The system shall automatically categorize transactions based on merchant data. [cite_start]The user shall be able to manually re-categorize individual transactions. [cite: 257]

## [cite_start]10.2 Income vs. Expenses (FR-STAT-002) [cite: 258]
* [cite_start]A line or bar chart shall show monthly income versus expenses over the selected period. [cite: 259]
* [cite_start]The dashboard shall display net savings per month and a cumulative savings trend line. [cite: 260]

## [cite_start]10.3 Account Balance Trends (FR-STAT-003) [cite: 261]
* [cite_start]A time-series chart shall show the daily closing balance for each account over the selected period. [cite: 262]
* [cite_start]The user shall be able to toggle visibility of individual accounts on the chart. [cite: 263]

## [cite_start]10.4 Top Recipients and Merchants (FR-STAT-004) [cite: 264]
